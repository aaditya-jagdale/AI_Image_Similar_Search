## AI Image Similar Search

This repository contains a small end‑to‑end prototype for **image similarity search** built on top of:

- **FastAPI** for the HTTP API
- **PyTorch / torchvision (ResNet‑50)** for image embeddings
- **ChromaDB** for vector storage and similarity search

You can:

- Ingest a catalog of textile images into a vector database.
- Run a FastAPI server that exposes a `/api/v1/search` endpoint.
- Use a simple HTML page to visually test similar‑image queries.

---

### Project structure

- `main.py` – FastAPI application (API entrypoint).
- `search_image.py` – Loads the ResNet‑50 model and performs vector search in ChromaDB.
- `ingest.py` – One‑off script to read extracted PDF data, compute embeddings, and populate the ChromaDB collection.
- `data/` – Local data directory.
  - `extract_data.py` – Utility to extract data from PDFs into JSON (used by `ingest.py`).
  - `output/` – Expected to contain:
    - `pdf_extracted_data.json` – Metadata for each textile design (generated by `extract_data.py`).
    - Image files referenced in the JSON under `image_path`.
- `chroma_db/` – On‑disk ChromaDB persistent store (created at runtime).
- `test_api.html` – Lightweight browser UI to upload an image and see similar matches.
- `dockerfile` – Container image definition for running the API.
- `Procfile` – Process definition suitable for platforms like Heroku / Render.
- `requirements.txt` – Python dependencies (generated from a working environment).

---

### Requirements

- Python **3.12** (matching the `dockerfile` base image).
- `pip` and a working C/C++ build toolchain (some dependencies may require compilation).
- Sufficient disk space for the ChromaDB store and your images.
- A GPU is **not required**, but having one will speed up embedding generation.

### Installation (local development)

1. **Clone the repository**

   ```bash
   git clone https://github.com/<your-username>/AI_Image_Similar_Search.git
   cd AI_Image_Similar_Search
   ```

2. **Create and activate a virtual environment (recommended)**

   ```bash
   python -m venv .venv
   source .venv/bin/activate  # on Windows: .venv\Scripts\activate
   ```

3. **Install dependencies**

   ```bash
   pip install --upgrade pip
   pip install -r requirements.txt
   ```

> **Note**: `requirements.txt` was generated from a working environment and may contain more packages than strictly necessary. If you want to slim it down, you can do a clean install in a fresh environment and progressively remove unused packages.

### Preparing the data

The ingestion workflow assumes:

- A JSON file at `data/output/pdf_extracted_data.json` (containing items with fields like `design_no`, `image_path`, `source_pdf`, etc.).
- Corresponding image files under `data/output/` whose paths match the `image_path` entries in the JSON.

If you already have this JSON and images, place them under `data/output/` as described above.

If you want to regenerate the JSON yourself, review `data/extract_data.py` and adapt it to your own PDF / data source. The exact format expected by `ingest.py` is:

```json
{
  "design_no": "UNIQUE_ID",
  "image_path": "relative/path/to/image.png",
  "source_pdf": "source.pdf",
  "width": "...",
  "stock": "...",
  "gsm": "..."
}
```

---

### Building the vector index (ingestion)

Once you have `pdf_extracted_data.json` and the corresponding images:

```bash
python ingest.py
```

What this does:

- Loads a pretrained **ResNet‑50** model with ImageNet weights.
- Crops the bottom‑right `500x500` region of each image.
- Computes an embedding for each image.
- Stores the embedding and metadata into a ChromaDB collection named `textile_images` under the `chroma_db/` directory.

You can control how many items are processed by tweaking the `LIMIT` constant in `ingest.py`.

### Running the API server

After ingestion completes (and `chroma_db/` has data), you can start the FastAPI server.

#### Using `uvicorn` directly

```bash
uvicorn main:app --reload --host 0.0.0.0 --port 8000
```

This will:

- Serve the simple test UI at `GET /` (which returns `test_api.html`).
- Expose the search endpoint at `POST /api/v1/search`.
- Serve images under `/media` from `data/output/` if that directory exists.

Navigate to `http://localhost:8000/` in your browser to use the test UI. Upload an image and the UI will display similar matches and their metadata.

#### Using `uvicorn` via Procfile (Heroku / Render style)

The supplied `Procfile` defines:

```bash
web: uvicorn main:app --host 0.0.0.0 --port $PORT
```

Most PaaS providers will read this file and start the app automatically.

### Running via Docker

You can build and run the project using the provided `dockerfile`.

1. **Build the image**

   ```bash
   docker build -t ai-image-similar-search -f dockerfile .
   ```

2. **Run the container**

   Make sure your `data/` and `chroma_db/` directories are available on the host. Mount them into the container so that:

   - The API can read your images and JSON.
   - ChromaDB can persist its index across container restarts.

   ```bash
   docker run --rm -p 8000:8000 \
     -v "$(pwd)/data:/app/data" \
     -v "$(pwd)/chroma_db:/app/chroma_db" \
     ai-image-similar-search
   ```

Then open `http://localhost:8000/` in your browser.

### API overview

- **`GET /`**
  - Returns the `test_api.html` file (simple browser UI).

- **`GET /api/v1/`**
  - Health‑check / status endpoint. Returns a small JSON payload.

- **`POST /api/v1/search`**
  - Request: multipart form with a single file field named `image_file`.
  - Behavior:
    - Reads the uploaded image.
    - Computes a ResNet‑50 embedding.
    - Queries ChromaDB for the 5 nearest neighbors.
  - Response: JSON array of matches with `id`, `distance`, and `metadata` fields.

### Development notes

- The ResNet‑50 instance is created once at import time (in `search_image.py`) and reused across requests.
- ChromaDB is configured to use a **persistent** store at `chroma_db/`. This directory will be created automatically the first time it is needed.
- The current preprocessing pipeline:
  - Resizes images to `256` pixels on the shortest side.
  - Center‑crops to `224×224`.
  - Applies ImageNet normalization.
  - Uses a `500×500` crop from the bottom‑right corner before feeding into the model.

If you change this pipeline, remember to **re‑ingest** the dataset so that stored embeddings are consistent with your new preprocessing.
